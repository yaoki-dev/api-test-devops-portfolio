## 要件まとめ

### 1.1 ビジネス要件（Why）

**課題**:
- Week 1-6の学習進捗とポートフォリオ実装の記録が手動で非効率
- 日次・週次の振り返りに時間がかかる（30-60分/回）
- 学習計画（10週ハイブリッドプラン）との乖離が見えづらい

**目標**:
- 学習記録時間を90%削減: 30分 → 3分
- 実装記録時間を95%削減: 20分 → 1分（「実装記録」入力のみ）
- 週次振り返り時間を70%削減: 60分 → 18分（自動集計）
- 進捗可視化: ポートフォリオメトリクスのリアルタイム更新

### 1.2 機能要件（What）

**必須機能（Phase 1）**

| 機能ID | 機能名 | 説明 | 優先度 |
|------|--------|------|-------|
| F1 | 学習開始ガイド | 現在のWeek/Day/Phase表示、推奨学習タスク提示 | Critical |
| F2 | 実装開始ガイド | ポートフォリオメトリクス表示、進行中タスク提示 | Critical |
| F3 | 学習記録自動化 | 学習時間・内容・課題の構造化記録 | Critical |
| F4 | 実装記録完全自動化 | Git履歴から実装内容・時間・メトリクス自動抽出 | Critical |
| F5 | 週次振り返り自動化 | 日次記録集計、週次サマリー自動生成 | High |

**推奨機能（Phase 2 - Optional）**

| 機能ID | 機能名 | 説明 | 優先度 |
|------|--------|------|-------|
| F6 | 学習計画調整支援 | 進捗遅延時に計画調整案を自動提案 | Medium |
| F7 | 目標達成予測 | 現在のペースでの目標達成日を予測 | Low |

### 1.3 非機能要件

| 要件カテゴリ | 要件内容 | 目標値 |
|---------|----------|--------|

**🆕 v2.1: ステータス判定ロジック（3段階簡素化）**

```python
def determine_status(mastery_level: int) -> str:
    """
    簡素化された3段階判定:
    - 85%以上 = 完了
    - 1~84% = 要復習
    - 0% = 未学習
    """
    if mastery_level >= 85:
        return "完了"
    elif mastery_level > 0:
        return "要復習"
    else:
        return "未学習"
```

**🆕 v2.1: 習熟度計算例**

```python
def update_mastery_level(
    old_mastery: int,           # 現在の累積習熟度 (e.g., 0%)
    actual_hours: float,        # 今日の学習時間 (e.g., 5h)
    estimated_hours: float,     # 推定必要時間 (e.g., 6h)
    target_level: int = 85      # 🆕 v2.1: 固定85%
) -> int:
    """
    累積習熟度の更新計算
    - 今日の学習による習熟度増分を算出
    - 既存の累積習熟度に加算（目標レベルを超えない）
    """
    # 今日の学習による習熟度増分
    mastery_gain = (actual_hours / estimated_hours) * target_level

    # 累積習熟度更新（目標レベル超過防止）
    new_mastery = min(target_level, old_mastery + mastery_gain)

    return int(new_mastery)

# 計算例
old = 0      # 初回学習前
actual = 5   # 今日5時間学習
estimated = 6  # 推定6時間必要
result = update_mastery_level(old, actual, estimated)
# → (5/6) * 85 = 70.8% → 71%

# ステータス判定
status = determine_status(71)
# → "要復習" (1-84%の範囲)
```

### 2.3 トリガー仕様

#### トリガー1: 学習開始

**検出キーワード**: "学習開始", "学習スタート", "今日の学習"

**実行フロー**:
1. Read learning_state.yaml → 現在状態取得
2. Read 10週ハイブリッドプラン → Week 7 Day 3セクション抽出
3. 提示:
   ```
   📍 現在位置: Week 7 Day 3 - Phase A (45%完了)
   ⚠️ 前回の課題: [issues リスト]
   🎯 推奨タスク: [next_learning_task]

   選択肢:
   a) 推奨タスク開始
   b) 前回課題復習
   c) 週次振り返り
   d) 学習計画調整
   ```

**所要時間**: 10秒

**異常系フロー**:

1. **learning_state.yaml読み込みエラー**:
   - `FileNotFoundError` → 初回セットアップフロー案内（学習開始と同様）
   - `yaml.YAMLError` → エラー行番号表示 + 手動修正案内
   - `PermissionError` → ファイル権限確認手順（`chmod 644 learning_state.yaml`）

2. **学習計画ファイル読み込みエラー**:
   - `FileNotFoundError` (10週ハイブリッドプラン.md) → 学習計画ファイル未配置警告
   - 該当Week/Dayセクション不在 → 汎用推奨タスク提示

3. **データ不整合エラー**:
   - `current_week`/`current_day` が null → デフォルト値（Week 1 Day 1）使用 + 警告表示
   - `current_phase` 不明 → 学習計画から推測 + 確認プロンプト

**復旧手順**:
- エラー発生時はlearning_state.yamlを変更せず既存状態保持
- ユーザーに手動確認・修正を促し、修正後に再実行案内

   - `FileNotFoundError` → 初回セットアップフロー案内
   - `yaml.YAMLError` → エラー箇所表示 + 手動修正案内
   - `PermissionError` → ファイル権限確認手順（`chmod 644 learning_state.yaml`）
   - YAML書き込み失敗 → ロールバック（既存ファイル保持）+ 再試行案内

2. **データ不整合エラー**:
   - `current_learning_item.skill_key` がnull/存在しない → 警告 + 手動skill_key入力プロンプト
   - `estimated_hours` = 0 → division by zero防止（mastery_level強制的に0%、ステータス「未学習」）
   - `actual_hours` 負値/異常値 → 0時間として扱う + 警告表示
   - `mastery_level` 計算結果100%超 → 100%にキャップ

3. **dual-file更新エラー（learning_state.yaml + daily_progress.md）**:
   - learning_state.yaml成功 + daily_progress.md失敗 → YAMLコミット、進捗ファイル警告表示
   - learning_state.yaml失敗 → 両ファイル更新中止、既存状態保持
   - daily_progress.md不在 → 警告表示 + 進捗記録スキップ（YAML更新は継続）

4. **ステータス判定エラー**:
   - `mastery_level` 異常値（負値/null） → デフォルトステータス「未学習」
   - skill_mastery[skill_key] 不在 → 新規スキル項目作成 + 警告表示

5. **learning_history追加エラー**:
   - 履歴配列書き込み失敗 → 警告表示（skill_mastery更新は成功として扱う）
   - 履歴10件制限超過 → 最古1件削除後追加

**復旧手順**:
- エラー発生時はlearning_state.yamlを変更せず既存状態保持（Write-Ahead Logging原則）
- dual-file更新の場合、learning_state.yaml優先（Single Source of Truth維持）
- ユーザーに手動確認・修正を促し、修正後に再実行案内

## データ整合性保証メカニズム

### 概要

本システムは複数ファイルへの更新を伴うため、データ整合性保証が必須です。**Single Source of Truth原則** に基づき、`learning_state.yaml` を権威ソースとし、`daily_progress.md` を副次的レポートファイルとして位置付けます。

### 1. Single Source of Truth 原則

**権威ソース（Authoritative Source）**: `learning_state.yaml`
- 全学習状態の正式記録
- 習熟度計算の基準データ
- 週次・日次状態の唯一の真実
- ステータス判定（完了/要復習/未学習）の根拠

**副次ソース（Secondary Source）**: `daily_progress.md`
- 可読性重視のレポートファイル
- ユーザー振り返り用テキスト記録
- YAML情報の人間可読形式展開
- 欠損・破損時はYAMLから再生成可能

**コンフリクト解決ポリシー**:
- YAML vs Markdown に不整合がある場合 → **YAML値を正とする**
- Markdown更新失敗 → YAML更新は継続（警告表示のみ）
- YAML更新失敗 → 全更新をロールバック（状態保存優先）

### 2. Write-Ahead Logging (WAL) パターン

**目的**: ファイル更新失敗時の状態復旧可能性確保

**実装方針**:

#### 2.1 更新前バックアップ
```yaml
# 更新フロー例（Trigger 3: 学習記録）
1. learning_state.yaml → メモリ読み込み
2. 新規learning_history作成（メモリ上）
3. 新規mastery_level計算（メモリ上）
4. バリデーション実行（計算値の妥当性確認）
5. ✅ バリデーション合格 → ファイル書き込み実行
6. ❌ バリデーション失敗 → エラー表示、ファイル更新せず
```

#### 2.2 書き込み順序の固定化
```
原則: 権威ソース（YAML）→ 副次ソース（Markdown）

理由:
- YAML更新失敗時 → 全更新中止（状態保存優先）
- YAML更新成功 + Markdown失敗 → YAML更新は維持、Markdown警告表示
```

#### 2.3 ロールバック戦略
```yaml
# YAML更新失敗時
エラー検出:
  - yaml.YAMLError（構文エラー）
  - PermissionError（書き込み権限なし）
  - OSError（ディスクフル等）

ロールバック実行:
  - メモリ上の変更を破棄
  - 既存ファイル保持（更新前状態維持）
  - エラー詳細 + 復旧手順をユーザーに提示
  - 代替手段（手動編集・再試行）案内
```

### 3. 2-Phase Commit パターン（デュアルファイル更新）

**適用トリガー**: Trigger 3（学習記録）、Trigger 4（実装記録）、Trigger 5（週次振り返り）

#### Phase 1: 検証フェーズ（Validation Phase）
```yaml
目的: 両ファイルが更新可能か事前確認

検証項目:
  1. learning_state.yaml:
     - ファイル存在確認
     - 書き込み権限確認
     - YAML構文妥当性確認（既存内容）
     - 計算値バリデーション（mastery_level 0-100範囲等）

  2. daily_progress.md:
     - ファイル存在確認
     - 書き込み権限確認
     - 挿入位置特定可能性確認（Week 7セクション存在等）

判定:
  - ✅ 両ファイル検証合格 → Phase 2へ進行
  - ❌ いずれか検証失敗 → 全更新中止、エラー表示
```

#### Phase 2: コミットフェーズ（Commit Phase）
```yaml
更新順序（固定）:
  1. learning_state.yaml 更新実行
     - 失敗時 → 即座に全更新中止、ロールバック
     - 成功時 → 2へ進行

  2. daily_progress.md 更新実行
     - 失敗時 → 警告表示、YAML更新は維持（Single Source of Truth優先）
     - 成功時 → 完全成功、完了メッセージ表示

部分失敗ハンドリング:
  - YAML成功 + Markdown失敗:
    - 状態: YAML更新済み、Markdown未更新
    - 対応: 警告表示「Markdownファイルへの記録は失敗しましたが、learning_state.yamlは正常更新されました」
    - 復旧: 次回トリガー実行時に自動補完、または手動でMarkdown編集
    - 影響: 最小限（権威ソースは更新済み）
```

### 4. アトミック性保証（Atomicity）

**目標**: 1トリガー実行 = 1アトミック操作

#### 4.1 トランザクション境界
```yaml
トリガー実行 = 1トランザクション:
  開始: ユーザーがトリガーキーワード入力
  終了: 「記録完了✅」メッセージ表示 or エラー表示

中間状態の回避:
  - 「learning_state.yamlは更新したがhistory未追加」状態を作らない
  - 全計算完了後に一括更新（部分更新禁止）
```

#### 4.2 並行実行制御
```yaml
想定シナリオ:
  - ユーザーが短時間に複数トリガー実行（学習記録 → 即座に実装記録）

制御方針:
```

```python
def validate_git_metrics(
    files_changed: int,
    lines_added: int,
    lines_deleted: int,

) -> bool:
```
