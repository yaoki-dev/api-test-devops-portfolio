## 🎯 4つの重要質問

これらの質問に回答することで、最適な統合オプションが明確になります：

### Q1: Task Masterのどの機能を使いたいか？

**回答オプション**:
- [ ] A. サブタスク階層（1.1, 1.2.1等）
- [ ] B. AI生成タスク（parse-prd, expand）
- [ ] C. 複雑性分析（analyze-complexity）
- [ ] D. MCPサーバー統合（Claude Code連携）
- [ ] E. タスク依存関係管理（v3.0にも既存）
- [ ] F. メトリクス可視化（v3.0にも既存）
- [ ] G. 特になし（v3.0で十分）

**判定ロジック**:
- **A/B/C/D のいずれか選択** → Option A/B/C検討対象
- **E/F のみ選択** → v3.0で既に実現可能 → Option D推奨
- **G を選択** → Option D一択

---

### Q2: シンプルさと高機能性のどちらを優先するか？

**回答オプション**:
- [ ] A. **シンプルさ最優先**（学習コスト最小化、保守性重視）
- [ ] B. **バランス重視**（必要最小限の機能追加、適度な複雑性許容）
- [ ] C. **高機能性優先**（多機能・統合、複雑性許容）

**判定ロジック**:
- **A を選択** → Option D推奨
- **B を選択** → Option C検討（費用対効果分析必須）
- **C を選択** → Option A/B検討（警告: YAGNI違反リスク高）

---

### Q3: 年間保守コストの許容範囲は？

**回答オプション**:
- [ ] A. **5時間以下**（現状維持、最小限の保守）
- [ ] B. **10-20時間**（軽微な機能追加、適度な保守）
- [ ] C. **50-80時間**（大規模統合、定期的な保守）

**判定ロジック**:
- **A を選択** → Option D一択
- **B を選択** → Option C検討可能
- **C を選択** → Option A/B検討可能（要慎重判断）

---

### Q4: 実装タイミングは？

**回答オプション**:
- [ ] A. **今すぐ実装必須**（プロジェクト要件）
- [ ] B. **試験的に試したい**（POC・評価目的）
- [ ] C. **後日検討**（優先度低）
- [ ] D. **実装不要**（現状で十分）

**判定ロジック**:
- **A を選択** → 緊急性確認 → Q1-Q3の回答に基づく
- **B を選択** → Option C の5h POC推奨
- **C を選択** → Option D継続 + 要件明確化待ち

## ❓ FAQ（よくある質問）

### Q: v3.0とTask Masterの主な違いは？

**A**:

| 側面 | v3.0 | Task Master |
|------|------|-------------|
| 哲学 | シンプル、個人開発向け | エンタープライズ級、複雑性許容 |
| タスク管理 | 単層（task_key） | 多層階層（1.1.1等） |
| AI統合 | なし | AI生成タスク、複雑性分析 |
| 設定管理 | YAML（allowed_tasks.yml） | JSON（tasks.json） |
| コード量 | 37行（todo.py） | 2000+行（複数モジュール） |
| セキュリティ | ホワイトリスト単層 | 多層検証 |

**結論**: v3.0は「シンプルかつ安全」、Task Masterは「高機能かつ複雑」。

---

### Q: 統合しないとどんな機能が使えないのか？

**A**: 統合しない場合（Option D）でも、**以下の機能は既に実現可能**:

✅ **v3.0で既に可能**:
- タスク依存関係（topological sort実装済み）
- メトリクス収集（JSON metadata記録）
- セキュリティ（allowed_tasks.ymlホワイトリスト）
- 自動実行（cron統合）
- 進捗トラッキング

❌ **統合しないと使えない機能**:
- サブタスク階層（1.1, 1.2.1等の多段階管理）
- AI生成タスク（parse-prd, expand, analyze-complexity）
- MCPサーバー統合（Claude Code連携、100+ツール）
- Task Master専用コマンド（task-master CLI）

**判断基準**: 「使えない機能」が年間20-80hの保守コストに見合うか？

---

### Q: Option Cの「API統合」は本当に安全か？

**A**: **はい、設計上安全です**。ただし、以下の条件付き：

**安全性を保証する設計**:
1. **Read-only Adapter**: Task Masterのtasks.jsonは読み取り専用
2. **2段階検証**: 形式検証 → ホワイトリスト検証 → ソース別検証
3. **Fail-secure**: エラー時はデフォルトで拒否
4. **shell=False**: subprocess.run()でshell injection防止
5. **セキュリティ監査ログ**: 全検証イベントを記録

**リスク**: 実装ミスによる脆弱性（人的エラー）

**対策**:
- `03_実装ガイド.md`のコードテンプレートを厳密に使用
- セキュリティテスト必須実行（test_security.py）
- コードレビュー実施

---

### Q: POC（概念実証）をどう実施すべきか？

**A**: **5時間のPOC実施推奨**（Option C試験導入）:

**ステップ1: 環境準備（1h）**
```bash
# 新規ブランチ作成
git checkout -b poc/task-master-integration

# Task Master初期化
task-master init
```

**ステップ2: 最小限実装（3h）**
- task_master_adapter.py作成（読み取り専用）
- security_validator.py作成（2段階検証）
- 単体テスト実施

**ステップ3: 評価（1h）**
- 使い勝手評価（タスク読み取り、実行フロー）
- 保守性評価（コード可読性、テスト容易性）
- 意思決定: 本実装 or 破棄

**評価基準**:
- ✅ 使い勝手が期待通り → 本実装検討（追加5-7h）
- ❌ 期待値未達 → `git branch -D poc/task-master-integration` で破棄

---

### Q: 既にTask Masterを導入済みの場合は？

**A**: **Option D（完全分離）を継続推奨**:

**理由**:
- Task Masterは独立したプロジェクトで十分に機能する
- v3.0とTask Masterは異なる哲学・用途
- 統合による複雑性増加のメリットが不明確

**推奨構成**:
```
~/projects/
├── my-todo-v3/          # v3.0（個人タスク管理）
│   ├── todo.md
│   ├── todo.py
│   └── allowed_tasks.yml
│
└── team-project/        # Task Master（プロジェクト管理）
    ├── .taskmaster/
    │   └── tasks/tasks.json
    └── README.md
```

**使い分け**:
- **v3.0**: 日常の個人タスク、シンプルなワークフロー
- **Task Master**: 大規模プロジェクト、チーム協働、複雑な依存関係
