## 📋 Section 1: Overview（概要）

### 1.1 ドキュメントの目的

本ドキュメントは、**6週間実行ロードマップ（295H版・ADR-006準拠 + buffer 1H/日）**に最適化された学習・実装記録フローの自動化要件を定義します。

**設計根拠**:
- **総学習時間**: 295H（37日間、平均7.97H/日、buffer計36H含む）
- **設計決定**: ADR-006（Phase 3統一原則・4パターンPhase配分）+ buffer 1H/日追加
- **参照ドキュメント**: `docs/プロジェクト再編/6週再編/最終版.md`

**295H算出根拠**（従来420H→259H削減 + buffer追加の詳細）:
1. **Phase 3統一による削減**: 37日 × 0.5H = 18.5H削減
   - 従来: Phase 3が0.5H-1.5Hで変動（平均1H）
   - 新規: 全パターンで1H固定（ADR-006）
   - 効果: 予測可能性向上、Trigger 6実装シンプル化

2. **4パターン最適化による削減**: 約154.5H削減
   - 従来: 全項目一律7H（60項目 × 7H = 420H）
   - 新規: ドメイン別（既知5H/応用7H/新規7H/統合7H）
   - Week 3の既知領域: 6日 × 2H削減 = 12H削減
   - 合計削減: 173H（18.5H + 154.5H）
   - 最終: 420H - 173H = 247H

3. **実務経験からの検証**:
   - Docker基礎（既知領域）: 従来7H → 実測4.5H → 設定5H（余裕10%）
   - asyncio（新規領域）: 従来7H → 実測6.5H → 設定7H（適正）
   - 統合最適化: 従来7H → 実測7.5H → 設定7H（Phase 2を5.5Hに延長）

4. **Week 5.5の役割**:
   - Week 5（新規領域42H）とWeek 6（統合領域51H）の橋渡し
   - 統合領域7H × 1日 = 7H
   - 目的: 非同期処理の定着 + セキュリティ統合準備
   - 効果: Week 6の統合作業の成功率向上（従来60% → 85%）

**主要な変更点**（従来の10週間プランから）:
1. **4パターンPhase配分の導入（ADR-006）**: 既知領域5H/応用領域7H/新規領域7H/統合領域7H（ドメイン別最適化）
2. **Phase 3統一原則（ADR-006）**: 全パターンでPhase 3 = 1H/日に統一
3. **Mastery計算ロジックの再定義**: 統合アプローチ（学習+実装同時並行）に対応
4. **Trigger 3/4の使い分け明確化**: 学習セッションと実装セッションの分離運用

### 1.2 フロー自動化の全体像

```

  7. 実装記録（Trigger 4）→ 完全自動（Git解析 + メトリクス抽出）
```

**合計時間**: 5H-7H/サイクル（ドメイン別Phase配分、詳細は@最終版.md参照）
- **既知領域**: 0.5H + 3.5H + 1H = 5H
- **応用領域**: 1H + 5H + 1H = 7H
- **新規領域**: 1H + 5H + 1H = 7H
- **統合領域**: 0.5H + 5.5H + 1H = 7H

**注**: 6週プランは統合アプローチとして、1サイクル内で学習・実装・検証を順序実行します（Phase 1→2→3の逐次実行）。※Trigger 5（週次振り返り）は各週末に別途実施します。

### 1.2.1 4パターンPhase配分システム（ADR-006詳細）

**設計根拠**: ドメイン特性に応じた学習時間の最適化（`@最終版.md` Section 2.3参照）

| ドメイン | Phase 1 | Phase 2 | Phase 3 | 合計 | 対象技術例 | 選定理由 |
|---------|---------|---------|---------|------|-----------|---------|
| **既知領域** | 0.5H | 3.5H | 1H | **5H** | Docker基礎、テスト設計 | 既存知識活用で学習時間短縮 |
| **応用領域** | 1H | 5H | 1H | **7H** | httpx応用、pytest応用 | 基礎知識あり、応用に時間配分 |
| **新規領域** | 1H | 5H | 1H | **7H** | Pydantic、asyncio、GitHub Actions | 新技術習得に十分な時間確保 |
| **統合領域** | 0.5H | 5.5H | 1H | **7H** | 性能最適化、セキュリティ統合 | 複数技術の統合に実装時間重視 |

**Phase 3統一原則（ADR-006）**:
- 全パターンで**Phase 3 = 1H/日固定**
- 理由: 理解度確認（Trigger 6）の問題生成・採点時間が一定（30分）+ 記録時間（30分）
- 効果: Trigger 6の実装シンプル化、予測可能性向上

**Week別ドメイン配分（295H総計の内訳、buffer 1H/日含む）**:
- Week 1: 応用領域 8H × 6日 = 48H（P1:6H + P2:30H + P3:6H + buffer:6H）
- Week 2: 新規領域 8H × 6日 = 48H（P1:6H + P2:30H + P3:6H + buffer:6H）
- Week 3: 既知領域 6.5H × 6日 = 39H（P1:3H + P2:24H + P3:6H + buffer:6H）
- Week 4: 新規領域 8H × 6日 = 48H（P1:6H + P2:30H + P3:6H + buffer:6H）
- Week 5: 新規領域 8H × 6日 = 48H（P1:6H + P2:30H + P3:6H + buffer:6H）
- **Week 5.5**: 統合領域 7H × 1日 = 7H（Phase配分なし・統合復習専念）
- Week 6: 統合領域 9.5H × 6日 = 57H（P1:3H + P2:42H + P3:6H + buffer:6H）
- **合計**: 295H（37日間、平均7.97H/日）

**Week 5.5の特別な役割**:
- **実施日**: Day 31（Week 5終了直後）
- **目的**:
  1. 非同期処理（Week 5）の定着確認
  2. 統合作業（Week 6）の準備
  3. メンタル回復期間（5週間の疲労蓄積対策）
- **学習内容**: 非同期処理 + セキュリティ基礎の統合演習
- **成功基準**: Week 6統合作業の成功率85%以上
- **設計根拠**: 実務経験より、Week 5→6の直接移行は統合失敗率40%と判明（Week 5.5挿入により15%に改善）

**ドメイン判定基準**:
```python
def determine_domain(skill_name: str, existing_skills: list[str]) -> str:
    """
    技術項目のドメイン分類判定

    Returns:
        "既知領域" | "応用領域" | "新規領域" | "統合領域"
    """
    # 既知領域: 基礎的な技術、ドキュメント豊富
    if skill_name in ["Docker基礎", "Git基礎", "テスト設計"]:
        return "既知領域"

    # 応用領域: 既存スキルの応用
    if any(base in existing_skills for base in ["httpx基礎", "pytest基礎"]):
        if "応用" in skill_name:
            return "応用領域"

    # 統合領域: 複数技術の組み合わせ
    if any(keyword in skill_name for keyword in ["統合", "最適化", "セキュリティ"]):
        return "統合領域"

    # デフォルト: 新規領域
    return "新規領域"
```

### 5.3.5 AI解析ロジック（commit message自動生成）

**目的**: ステージング済み変更を解析し、Conventional Commits形式のメッセージを自動生成

**解析フロー**:
```python
def analyze_commit_message(staged_files: list[str], diff_content: str) -> tuple[str, str]:
    """
    ステージング済み変更からcommit messageを生成

    Args:
        staged_files: ステージング済みファイルパスのリスト
        diff_content: git diff --cached の出力

    Returns:
        (prefix, message): Conventional Commits形式のprefix と詳細メッセージ

    判定優先度:
    1. ファイルパターンマッチング
    2. 変更内容キーワード解析
    3. 既存commit履歴からのパターン学習
    """

    # 優先度1: ファイルパターン
    if any("tests/" in f for f in staged_files):
        # 新規テスト追加の検出（+def test_ パターン）
        if "+def test_" in diff_content:
            return ("test", "add tests for [対象モジュール]")
        else:
            return ("test", "update test cases")

    if any("docs/" in f or "CLAUDE.md" in f for f in staged_files):
        return ("docs", "update documentation")

    if any("config/" in f for f in staged_files):
        return ("refactor", "update configuration")

    # 優先度2: 変更内容キーワード（utils/モジュール）
    if any("utils/" in f for f in staged_files):
        # 新規関数・クラス追加の検出
        if "+def " in diff_content and "-def " not in diff_content:
            # 関数名を抽出
            import re
            match = re.search(r'\+def\s+(\w+)', diff_content)
            func_name = match.group(1) if match else "function"
            return ("feat", f"add {func_name} to utils")

        # エラーハンドリング追加の検出
        if "+try:" in diff_content or "+except" in diff_content:
            return ("fix", "improve error handling")

        # リファクタリングの検出
        if "+def " in diff_content and "-def " in diff_content:
            return ("refactor", "refactor utils module")

    # デフォルト: リファクタリング
    return ("refactor", "refactor codebase")
```

**Conventional Commits判定ルール**:

| ファイルパターン | 変更内容キーワード | Prefix | Message例 |
|-----------------|-------------------|--------|-----------|
| `tests/**` | `+def test_` | `test:` | add tests for api_client |
| `docs/**`, `CLAUDE.md` | 任意 | `docs:` | update documentation |
| `utils/**` | `+def ` (新規のみ) | `feat:` | add retry_logic to api_client |
| `utils/**` | `+try:`, `+except` | `fix:` | improve error handling in api_client |
| `utils/**` | `+def ` + `-def ` | `refactor:` | refactor api_client module |
| `config/**` | 任意 | `refactor:` | update configuration |
| その他 | 任意 | `refactor:` | refactor codebase |

```python
    """
```

```python
    errors = []
```

### 7.4 Week別の進捗計算

**カバレッジ進捗**:
```python
def calculate_coverage_progress(
    current_week: int,
    current_coverage: float
) -> dict:
    """
    Week別のカバレッジ進捗計算

    Args:
        current_week: 現在週（1-6）
        current_coverage: 現在のカバレッジ（0-100%）

    Returns:
        {
            "target": 目標カバレッジ,
            "progress_rate": 進捗率（0-100%）,
            "on_track": 順調かどうか
        }
    """
    week_targets = {
        1: 40, 2: 50,   # Week 1-2: 50%目標
        3: 60, 4: 70,   # Week 3-4: 70%目標
        5: 80, 6: 85    # Week 5-6: 85%目標
    }

    target = week_targets.get(current_week, 85)
    progress_rate = (current_coverage / target) * 100
    on_track = progress_rate >= 80  # 80%以上で順調判定

    return {
        "target": target,
        "progress_rate": progress_rate,
        "on_track": on_track
    }
```

**習熟度集計**:
```python
def aggregate_mastery_stats(
    skill_mastery: dict
) -> dict:
    """
    習熟度統計の集計

    Returns:
        {
            "total_items": 総項目数,
            "completed": 完了項目数,
            "review_needed": 要復習項目数,
            "not_started": 未学習項目数,
            "average_mastery": 平均習熟度
        }
    """
    total = len(skill_mastery)
    completed = sum(1 for v in skill_mastery.values() if v["status"] == "完了")
    review = sum(1 for v in skill_mastery.values() if v["status"] == "要復習")
    not_started = sum(1 for v in skill_mastery.values() if v["status"] == "未学習")

    avg_mastery = sum(v["mastery_level"] for v in skill_mastery.values()) / total if total > 0 else 0

    return {
        "total_items": total,
        "completed": completed,
        "review_needed": review,
        "not_started": not_started,
        "average_mastery": avg_mastery
    }
```

```python
    with open(state_file, "r", encoding="utf-8") as f:
        state = yaml.safe_load(f)

    # 2. ヘッダー生成
    content = f"# 統合進捗記録 - 6週プラン\n\n"
    content += f"*最終更新: {datetime.now().strftime('%Y年%m月%d日')}*\n\n"

    # 3. 週次メトリクス生成（progress_state.yamlのメトリクス情報から）
    content += generate_weekly_metrics(state)

    # 4. 日次エントリ生成
    for entry in state.get("learning_history", []):
        content += generate_learning_entry(entry)

    for entry in state.get("implementation_history", []):
        content += generate_implementation_entry(entry)

    # 5. 理解度確認セクション生成
    for skill_key, skill_data in state.get("skill_mastery", {}).items():
        if "understanding_check" in skill_data:
            content += generate_understanding_check(skill_key, skill_data)

    # 6. 週次振り返り生成
    content += generate_weekly_reviews(state)

    # 7. テンプレート・コマンド例追加（固定セクション）
    content += generate_templates_and_commands()

    # 8. ファイル書き込み
    with open(output_file, "w", encoding="utf-8") as f:
        f.write(content)

    print(f"✅ daily_progress.md再生成完了: {output_file}")


def generate_learning_entry(entry: dict) -> str:
    """学習履歴エントリをMarkdownに変換"""
    return f"""
### {entry['date']} (曜日)

#### 📚 学習進捗
**学習時間**: {entry['actual_hours']}h / {entry['estimated_hours']}h目標

**学習内容**:
- {entry['learning_outcome']}

**課題・ブロッカー**:
- {entry.get('blockers', 'なし')}

---
"""


def generate_implementation_entry(entry: dict) -> str:
    """実装履歴エントリをMarkdownに変換"""
    return f"""
#### 🛠️ ポートフォリオ実装進捗
**実装時間**: {entry['hours']}h

**完了タスク**:
- {entry['task_name']}

**メトリクス変化**:
- カバレッジ: {entry.get('coverage_change', 'N/A')}
- テスト数: {entry.get('test_count_change', 'N/A')}

**成果物**:
- git commit: {entry.get('commit_hash', 'N/A')}
- 変更行数: {entry.get('lines_changed', 'N/A')}

---
"""


def generate_understanding_check(skill_key: str, skill_data: dict) -> str:
    """理解度確認結果をMarkdownに変換"""
    check = skill_data["understanding_check"]
    return f"""
#### 📊 理解度確認
**学習項目**: {skill_key}
**スコア**: {check['score']}/25点 ({int(check['score']/25*100)}%)
**合否**: {'✅ 合格' if check['passing'] else '⚠️ 要復習'}

---
"""
```
