## 2: 設計フェーズワークフロー

### `[WF-01]` 曖昧なアイデアの具体化

**シナリオ**: ユーザーから漠然とした「新しいアプリを作りたい」等のリクエストがあった場合。

**実行シーケンス**:

1. **`socratic-mentor`**: 対話を通じて、ユーザーが本当に解決したい課題や目的（Why）を掘り下げる。
   - **アウトプット**: 課題の明確化、解決すべきペインポイント定義
   - **成功基準**: Why（目的）が3段階以上深掘りされ、本質的課題が特定されている

2. **`requirements-analyst`**: 明確になった課題を基に、具体的な機能要件、非機能要件を定義し、ドキュメント化する。
   - **アウトプット**: 要件定義書（機能要件10項目、非機能要件5項目以上）
   - **成功基準**: SMART原則に基づいた要件定義が完了している

3. **`system-architect`**: 定義された要件を基に、実現可能性を評価し、大まかなシステム構成案を提示する。
   - **アウトプット**: システム構成図、技術スタック候補、リスク評価
   - **成功基準**: 3つ以上の実現可能な構成案が提示され、それぞれのトレードオフが明確

**次のステップ**: `[WF-02]`でアーキテクチャを詳細化、または`[WF-06]`でテスト戦略を並行策定

---

### `[WF-02]` システム全体のアーキテクチャ設計

**シナリオ**: 新規プロジェクトの全体設計、または大規模な改修する場合。

**実行シーケンス**:

1. **`system-architect`**: 全体の構成、技術スタック、コンポーネント間の境界を定義する。
   - **アウトプット**: システム全体図、技術スタック決定、コンポーネント分割戦略
   - **成功基準**: SOLID原則に基づいた設計、将来の拡張性を考慮

2. **`backend-architect` + `devops-architect`**: 各担当領域のアーキテクトが並行して、より詳細な設計する。
   - **並行実行**: 2エージェント同時実行（Task tool並列呼び出し）
   - **backend**: API設計、データモデル、認証・認可方式
   - **devops**: CI/CD戦略、インフラ構成、監視・ログ戦略
   - **成功基準**: 各領域で具体的な設計書が完成し、相互依存が明確化
   - **Note**: frontend-architectは削除済み。フロントエンド設計が必要な場合は`backend-architect`でAPI設計観点から対応

3. **`system-architect`**: 各設計を統合し、全体の整合性を確認する。
   - **アウトプット**: 統合アーキテクチャドキュメント、インターフェース定義
   - **成功基準**: コンポーネント間の依存関係が明確、技術的負債リスクが評価済み

**次のステップ**: `[WF-03]`でDevOps基盤設計、`[WF-04]`で実装開始、`[WF-06]`でテスト戦略策定

## 3: 開発基盤ワークフロー

### `[WF-03]` DevOps/CI/CDの全体戦略設計

**シナリオ**: プロジェクトの自動化された開発・運用基盤を設計する場合。

**実行シーケンス**:

1. **`devops-architect`**: CI/CD、コンテナ戦略、監視、ロギングの全体方針を策定する。
   - **アウトプット**: DevOps戦略書、CI/CDパイプライン設計、監視・ログ戦略
   - **成功基準**: 開発→本番デプロイまでの自動化フローが明確

2. **`ci-cd-pipeline`**: パイプラインの具体的な実装を担当する。
   - **アウトプット**: GitHub Actions/GitLab CI設定、テスト自動化、デプロイスクリプト
   - **成功基準**: コミット→テスト→ビルド→デプロイが完全自動化

3. **`docker-devops`**: コンテナイメージの最適化と管理を担当する。
   - **アウトプット**: Multi-stage Dockerfile、docker-compose設定、イメージ最適化
   - **成功基準**: 本番用イメージが50MB以下、セキュリティスキャン合格

4. **`logging-monitoring`**: ログとメトリクスの収集・可視化基盤を実装する。
   - **アウトプット**: structlog設定、Prometheus/Grafana統合、アラート設定
   - **成功基準**: SLI/SLOに基づいたアラートが設定され、ダッシュボードが稼働

**次のステップ**: `[WF-05]`でコンテナ化実装、`[WF-08]`で障害対応手順整備

---

### `[WF-04]` 新規APIエンドポイント実装

**シナリオ**: APIの新規開発・実装する場合。

**実行シーケンス**:

1. **`backend-architect`**: APIの仕様（エンドポイント、リクエスト/レスポンス形式、認証方式）を設計する。
   - **アウトプット**: API設計書（OpenAPI 3.0仕様）
   - **成功基準**: エンドポイント定義、入出力スキーマ、エラーコードが明確

2. **`python-expert`**: 設計仕様書に基づき、Python（FastAPI等）でビジネスロジックを実装する。
   - **アウトプット**: APIエンドポイント実装コード、エラーハンドリング
   - **成功基準**: 仕様通りの実装が完了し、型ヒント・docstringが完備

3. **`api-testing`**: 実装されたAPIに対して、`pytest`を用いた単体・結合テストを作成し、品質を保証する。
   - **アウトプット**: pytestテストコード、カバレッジレポート
   - **成功基準**: カバレッジ85%以上、正常系・異常系網羅

4. **`ci-cd-pipeline`**: テストが通った実装をCI/CDパイプラインに統合し、自動デプロイの準備を整える。
   - **アウトプット**: パイプライン設定更新、自動デプロイ設定
   - **成功基準**: コミット→テスト→デプロイが自動化されている

**次のステップ**: `[WF-06]`でテスト戦略強化、`[WF-07]`でセキュリティ診断

---

### `[WF-05]` アプリケーションのコンテナ化

**シナリオ**: 開発したアプリケーションをDockerコンテナで実行可能にする場合。

**実行シーケンス**:

1. **`docker-devops`**: Multi-stage buildを用いて、開発用・本番用のDockerfileを最適化して作成する。
   - **アウトプット**: Multi-stage Dockerfile、.dockerignore、ビルド最適化
   - **成功基準**: イメージサイズ50MB以下、ビルド時間3分以内

2. **`security-testing`**（実行層）: 作成されたコンテナイメージに脆弱性がないかスキャンする。
   - **アウトプット**: Trivy/Grypeスキャンレポート、脆弱性対応計画
   - **成功基準**: Critical/High脆弱性0件

3. **`ci-cd-pipeline`**: コンテナビルドとプッシュをパイプラインに組み込む。
   - **アウトプット**: パイプライン設定更新、レジストリ連携
   - **成功基準**: コミット→ビルド→プッシュが自動化

**次のステップ**: `[WF-03]`でCI/CD統合、`[WF-09]`でパフォーマンス最適化

## 4: 品質保証ワークフロー

### `[WF-06]` プロジェクト全体のテスト戦略策定

**シナリオ**: プロジェクトの品質保証計画を立てる場合。

**実行シーケンス**:

1. **`quality-engineer`**: プロジェクト全体のリスクを評価し、どのようなテストをどの段階で行うべきかの戦略を策定する。
   - **アウトプット**: テスト戦略書、テストピラミッド定義、品質ゲート設定
   - **成功基準**: 単体/結合/E2E/性能/セキュリティの各テスト範囲が明確

2. **`api-testing` + `performance-testing`（実行層）+ `security-testing`（実行層）+ `qa-expert`**: 各専門エージェントが、戦略に基づいて具体的なテスト計画と実装を担当する。
   - **並行実行**: 4エージェント同時実行
   - **api-testing**: pytest単体・統合テスト実装
   - **performance-testing**（実行層）: 負荷テスト計画、ベンチマーク基準設定
   - **security-testing**（実行層）: OWASP Top10に基づく診断計画
   - **qa-expert**: E2Eテスト計画、Playwright MCP統合による自動検証
   - **成功基準**: 各専門領域でテスト計画書が完成

**次のステップ**: `[WF-07]`でセキュリティ診断実施、`[WF-09]`でパフォーマンステスト実施

---

### `[WF-07]` セキュリティ脆弱性診断

**シナリオ**: 定期的なセキュリティチェックや、リリース前の最終確認する場合。

**実行シーケンス**:

1. **`security-engineer`**（戦略層）: 既知の脅威モデルに基づき、診断すべき対象とスコープを定義する。
   - **アウトプット**: 診断計画書、脅威モデル、診断スコープ定義
   - **成功基準**: OWASP Top10カバレッジ100%の診断計画

2. **`security-testing`**（実行層）: 定義されたスコープに従い、自動スキャンツールや手動テストを用いて脆弱性を探索する。
   - **アウトプット**: 脆弱性診断レポート、CVSS評価、再現手順
   - **成功基準**: Critical/High脆弱性の全検出、False Positive除去

3. **`technical-writer`**: 発見された脆弱性と推奨される対策をまとめたレポートを作成する。
   - **アウトプット**: セキュリティレポート、対策優先度リスト
   - **成功基準**: 経営層・開発者の両方に理解可能なレポート

**次のステップ**: `[WF-12]`でアーキテクチャレビュー、脆弱性対応後は`[WF-06]`で回帰テスト

## 5: 運用・保守ワークフロー

### `[WF-08]` 本番障害の原因究明

**シナリオ**: 本番環境で発生した障害やエラーの原因を調査し、解決する場合。

**実行シーケンス**:

1. **`logging-monitoring`**: 障害発生時刻周辺のログ、メトリクス、トレースなど、分析に必要な客観的データをすべて収集・集約する。
   - **アウトプット**: ログ集約レポート、メトリクスグラフ、トレースデータ
   - **成功基準**: 障害前後30分のすべてのログ・メトリクスが収集済み

2. **`root-cause-analyst`**: 収集されたデータに基づき、仮説を立てて検証し、障害の根本原因を特定する。
   - **アウトプット**: 根本原因分析レポート、5 Whys分析、再発防止策
   - **成功基準**: 根本原因が特定され、再現手順が明確

3. **`python-expert`**: 特定された原因を解決するためのコード修正する。
   - **アウトプット**: バグフィックスコード、単体テスト
   - **成功基準**: 修正が最小限の変更で、既存機能に影響なし

4. **`debugger`**: エラー、テスト失敗、予期しない動作を体系的に調査し、根本原因を特定・修正する。
   - **アウトプット**: デバッグログ分析、問題特定レポート、修正検証
   - **成功基準**: 問題の再現手順が明確、根本原因が特定され修正済み

5. **`quality-engineer`**: 修正がデグレードを引き起こさないか、また障害が再発しないかを確認するためのテストを実施する。
   - **アウトプット**: 回帰テスト結果、デプロイ前検証レポート
   - **成功基準**: 全テスト合格、障害が再現しないことを確認

**次のステップ**: `[WF-09]`でパフォーマンス影響確認、`[WF-03]`で監視強化

---

### `[WF-09]` パフォーマンス劣化の原因調査

**シナリオ**: システムの応答が遅いなど、パフォーマンスに関する問題を調査・解決する場合。

**実行シーケンス**:

1. **`logging-monitoring`**: パフォーマンス劣化が疑われる期間のCPU使用率、メモリ使用量、API応答時間のP95メトリクスなどを収集する。
   - **アウトプット**: パフォーマンスメトリクス集約レポート、ベースライン比較
   - **成功基準**: 問題期間のすべての関連メトリクスが可視化

2. **`performance-engineer`**（戦略層）: 収集されたデータに基づき、どの部分にボトルネックがあるかを分析・特定し、改善策を立案する。
   - **アウトプット**: ボトルネック分析レポート、最適化提案（3案以上）
   - **成功基準**: ボトルネックが特定され、改善効果が定量的に予測

3. **`python-expert`**: 改善策をコードに反映させる。
   - **アウトプット**: 最適化コード、パフォーマンステスト
   - **成功基準**: 実装が完了し、ローカル環境でパフォーマンス改善を確認

4. **`performance-testing`**（実行層）: 修正後、再度負荷テストを行い、実際に性能が改善されたかを定量的に証明する。
   - **アウトプット**: ベンチマークレポート（before/after比較）
   - **成功基準**: 目標性能（例: P95レイテンシ50%削減）を達成

**次のステップ**: `[WF-03]`で監視ダッシュボード更新、`[WF-08]`で根本原因分析

## 6: プロジェクト管理ワークフロー

### `[WF-10]` タスク分解とコンテキスト管理

**シナリオ**: 複雑なプロジェクトや長期セッションで、タスクの最適粒度分解と文脈管理が必要な場合。

**実行シーケンス**:

1. **`context-manager`**: プロジェクト全体の目標とマイルストーンを定義し、セッション間の文脈連続性を確保する。
   - **アウトプット**: プロジェクト全体ロードマップ、マイルストーン定義、セッション管理計画
   - **成功基準**: 3-5個のマイルストーンが定義され、各マイルストーンの成功基準が明確

2. **`task-decomposition-expert`**: 各マイルストーンを実行可能な粒度のタスクに分解し、成功基準と検証方法を明確化する。
   - **アウトプット**: タスク分解ツリー、各タスクの成功基準、依存関係マップ
   - **成功基準**: 各タスクが1-3日で完了可能な粒度、成功基準が測定可能

3. **`context-manager`**: 分解されたタスクをセッション管理に統合し、進捗追跡とブロッカー管理する。
   - **アウトプット**: タスクトラッキングダッシュボード、ブロッカーリスト
   - **成功基準**: リアルタイム進捗可視化、ブロッカーが24時間以内に解消

4. **適切な専門Agent**: 分解されたタスクごとに最適なAgentを選択して実行する（例: python-expert、docker-devops等）。
   - **実行方法**: Agent Selection Strategy Mappingに基づいた動的選択
   - **成功基準**: 各タスクの成功基準達成

5. **`context-manager`**: タスク完了後、成果物と学習内容を記録し、次のセッションで参照可能にする。
   - **アウトプット**: セッションサマリー、学習ログ、次回セッション準備事項
   - **成功基準**: セッション再開時に前回の文脈が100%復元可能

**継続的適用**: プロジェクト全体で継続的に実施  
**連携ワークフロー**:  
- `[WF-01]`要件定義前: 曖昧な要求を具体的タスクへ分解する際の実施推奨
- `[WF-02]`アーキテクチャ設計前: 大規模設計タスクを段階的へ分解する際の実施推奨
- `[WF-04]`実装前: 複雑な実装タスクを分解する際の実施推奨
- `[WF-12]`レビュー前: レビュースコープを適切な粒度へ分解する際の実施推奨

### `[WF-11]` AI協働品質向上（プロンプトエンジニアリング）

**シナリオ**: AI依頼品質をLevel 2（基本的依頼）からLevel 3（技術的依頼）、Level 4（高度な依頼）へ段階的に向上させたい場合。

**実行シーケンス**:

1. **`prompt-engineer`**: 現在のAI依頼文を分析し、Level判定と改善点を特定する。
   - **アウトプット**: プロンプト品質評価レポート、Level判定（1-4）、改善ポイント
   - **成功基準**: 依頼文の曖昧さ・欠落要素が明確化され、改善方向性が提示

2. **`prompt-engineer`**: 改善されたプロンプトテンプレートを生成し、コンテキスト・制約・成功基準・技術要件を組み込む。
   - **アウトプット**: 改善プロンプトテンプレート、チェックリスト
   - **成功基準**: Level 3-4品質のプロンプトが生成され、技術要件が明確

3. **適切な専門Agent**: 改善されたプロンプトで実際のタスクを実行する（例: python-expert、docker-devops等）。
   - **実行方法**: Agent Selection Strategy Mappingに基づいた選択
   - **成功基準**: 高品質なプロンプトにより、1回目の出力で要件を満たす

4. **`code-reviewer`または`architect-review`**: 生成された成果物をレビューし、プロンプト品質が実際に向上したかを検証する。
   - **アウトプット**: 成果物品質評価、プロンプト改善効果測定
   - **成功基準**: 成果物品質が80%以上、手戻り回数が50%削減

5. **`prompt-engineer`**: レビュー結果を分析し、次回のプロンプト改善に活かすパターンを抽出する。
   - **アウトプット**: プロンプトパターンライブラリ、ベストプラクティス集
   - **成功基準**: 再利用可能なパターンが5個以上蓄積

**継続的改善**: プロンプト品質向上を習慣化  
**連携ワークフロー**:  
- **全ワークフロー開始前**: Agent依頼文の品質分析・改善に実施推奨  
- `[WF-01]`要件定義時: 曖昧な要求を明確化するプロンプト最適化に実施推奨  
- `[WF-04]`実装依頼時: 技術要件を明確化したLevel 4プロンプト作成に実施推奨  
- `[WF-12]`レビュー前: プロンプト品質検証・レビュー基準明確化に実施推奨

       - 根拠: 業界標準（Google Testing Blog, Netflix基準）、実用的上限値
     - **Branch Coverage**: 80%以上（条件分岐の全パターン網羅）
       - 根拠: 条件分岐組み合わせ爆発を考慮した現実的閾値、Lineより5%低設定
     - **Path Coverage**: 75%以上（関数内の実行経路網羅）
       - 根拠: 複雑関数の全経路網羅は非現実的、優先経路カバー重視
     - **Function Coverage**: 90%以上（全関数のうちテスト実行された割合）
       - 根拠: 関数レベルは100%近く必須、未テスト関数は技術的負債
     - 測定ツール: pytest-cov, coverage.py, Branch coverage plugin
     - 参考文献: [Google Testing Blog - Code Coverage Best Practices](https://testing.googleblog.com/), Martin Fowler's "Test Coverage"

   - **成功基準**: 品質ゲート合格、レビュー指摘事項100%解決

### 実行時のベストプラクティス

- **エージェント選択**: Agent Selection Strategy Mappingの推奨エージェントを優先
- **文脈保持**: 長期ワークフローでは`[WF-10]`でセッション管理を並行実施

### メンテナンスガイド

#### 新規ワークフロー追加手順（WF-14以降）

新しいワークフローを追加する際は、以下のチェックリストに従ってください:

**Phase 1: ワークフロー設計**
- [ ] ワークフローIDを決定（例: [WF-14]）
- [ ] シナリオを明確化（どのような状況で使用するか）
- [ ] 実行シーケンスを定義（3-6ステップ推奨）
- [ ] 各ステップの成功基準を設定（測定可能な基準）
- [ ] 推定実行時間を見積もり（過去の類似ワークフローを参考）
- [ ] 主担当エージェント数をカウント

**Phase 2: ドキュメント更新**
- [ ] **Agent Selection Strategy Mapping**テーブルに新規行を追加
  - フェーズ、使用場面、主要エージェント、ワークフローID、目的と期待成果を記載
- [ ] **Complex Workflow Playbooksクイックリファレンス**テーブルに新規行を追加
  - ワークフローID、名称、シナリオ要約、主要フェーズ、メモリーセクションを記載
- [ ] **workflow_playbooks_guide**メモリーファイルを更新
  - 1: ワークフロー概要テーブルに新規行を追加
  - 該当セクション（2-7）に詳細な実行シーケンスを追加
  - "次のステップ"で関連ワークフローとの連携を明示

**Phase 3: 影響分析・テスト**
- [ ] 既存ワークフローの"次のステップ"に新ワークフローへのリンクを追加
- [ ] 新ワークフローが既存ワークフローと重複していないか確認
- [ ] テスト実行: 新ワークフローを実際のタスクで試行し、改善点を抽出
- [ ] ドキュメントのバージョン更新（最終更新日を変更）

**Phase 4: 品質検証**
- [ ] `technical-writer`でドキュメント品質レビュー実施
- [ ] `architect-review`でワークフロー整合性レビュー実施
- [ ] レビュー結果に基づき改善実施
- [ ] 最終承認取得

#### 既存ワークフロー改善手順

既存ワークフローに問題が発見された場合:

- [ ] 問題の種類を特定（曖昧な記述、ステップ不足、成功基準不明確等）
- [ ] 影響範囲を分析（該当ワークフローのみ、関連ワークフローも影響）
- [ ] 改善案を作成（具体的な修正内容を記載）
- [ ] 上記Phase 2-4のチェックリストを実施
- [ ] 変更履歴を記録（最終更新日、変更内容を明記）

#### ワークフロー削除・非推奨化手順

ワークフローが不要になった場合:

- [ ] 削除理由を文書化（廃止理由、代替ワークフローの提案）
- [ ] 影響分析: 該当ワークフローを参照している箇所をすべて特定
- [ ] 代替ワークフローへの移行ガイド作成
- [ ] Agent Selection Strategy Mappingから削除（または非推奨マーク追加）
- [ ] クイックリファレンステーブルから削除
- [ ] メモリーファイルから該当セクションを削除（または非推奨セクション移動）
- [ ] 変更履歴を記録
