## 1. 要件概要

### 1.1. ビジネス要件

**目的**: Task Masterのタスク管理システムに自動実行機能を追加し、深夜帯等のオフアワーに自動でコマンド実行・ステータス更新を行う。

**背景**:
- 現状: リグレッションテスト・セキュリティスキャン等を手動実行（週5h）
- 課題: 開発時間の圧迫、テスト忘れによる品質リスク
- 解決: 深夜帯自動実行により開発時間を確保、品質保証を継続的に実施

**成功基準**:
- 深夜帯（11:00-05:00 UTC+7）にタスクが自動実行されること
- 実行結果がTask Masterのステータスに反映されること
- セキュリティリスク（コマンドインジェクション）が0件であること
- 手動実行時間が週5h → 0.5hに削減されること

### 1.2. 機能要件

#### FR-1: 自動実行タスク定義
- タスクに`autoExecute`メタデータを追加可能
- スケジュール種別（nightly/hourly/daily/weekly/custom）を指定可能
- リトライ設定（有効/無効、最大回数）を指定可能
- **手動タスクは明示的に`"enabled": false`を設定（Pattern B推奨）**

#### FR-2: セキュリティ検証
- `allowed_tasks.yml`ホワイトリストによるコマンド検証
- `taskKey`による間接実行（直接コマンド指定禁止）
- Task ID別アクセス制御（ワイルドカード対応）

#### FR-3: スケジューリング
- 時間帯指定（startHour/endHour、深夜対応）
- 曜日指定（daysOfWeek: 0-6）
- 重複実行防止（最終実行時刻チェック）

#### FR-4: 実行管理
- タイムアウト制御（タスク別設定）
- エラーハンドリング（タイムアウト、コマンド失敗）
- リトライロジック（指数バックオフ）

#### FR-5: ステータス同期

**ステータス値の定義**:
| 日本語 | Task Master値 | 説明 | 使用場面 |
|--------|--------------|------|---------|
| 未着手 | `pending` | 実行前の初期状態 | 全タスク共通 |
| 進行中 | `in-progress` | 実行中 | 全タスク共通 |
| 完了 | `done` | 正常終了 | 全タスク共通 |
| エラー | `error` | 実行失敗 | **自動実行タスク専用** |

**Task Master CLIによるステータス更新**:
- 実行前: `pending` → `in-progress`
- 成功時: `in-progress` → `done`
- 失敗時: `in-progress` → `error`（**自動実行タスク専用**、手動タスクでは通常使用しない）

**自動実行タスクのステータス遷移フロー**:
```
pending → in-progress → done (exit_code == 0)
                     ↘ error (exit_code != 0)
```

**手動タスクのステータス遷移フロー**:
```
pending → in-progress → done
```

**注意事項**:
- `error`ステータスは自動実行時に自動的に設定される
- 手動タスクで`error`を設定することは技術的に可能だが、通常は使用しない
- 手動タスクのエラーは`done`後に別途記録することを推奨

#### FR-6: 実行履歴管理
- 最新10件の実行記録を保持（リングバッファ）
- メタデータ: timestamp, exitCode, duration, stdout/stderr
- JSON Lines形式でメトリクスログ記録

### 1.3. 非機能要件

#### NFR-1: パフォーマンス
- タスク抽出処理: 100ms以内（tasks.json 1000タスク時）
- セキュリティ検証: 10ms以内（1タスクあたり）
- ステータス更新: 5秒以内（Task Master CLI呼び出し）

#### NFR-2: 信頼性
- Cron起動失敗時の自動リカバリー（次回実行時に再試行）
- ログローテーション（10MB/ファイル、5世代保持）
- JSONパースエラー時の緊急停止

#### NFR-3: 保守性
- コード行数: 500行以内（auto_executor.py）
- クラス責務の単一化（SRP遵守）
- 型ヒント100%カバレッジ

#### NFR-4: セキュリティ
- コマンドインジェクション検出率: 100%
- subprocess.run()は常にshell=False
- ファイルパーミッション: 644 (設定), 755 (スクリプト)

**リトライロジック**:
```python
def execute_with_retry(task: Dict, task_config: Dict) -> Dict:
    """
    リトライ機能付き実行

    - maxRetries以内でリトライ
    - 指数バックオフ（2^retry_count秒待機）
    - retryOnFailure=falseならリトライしない
    """
    max_retries = task["autoExecute"].get("maxRetries", 3)
    retry_count = 0

    while retry_count <= max_retries:
        result = CommandExecutor().execute(task, task_config)

        if result["exit_code"] == 0:
            return result

        if not task["autoExecute"].get("retryOnFailure", True):
            return result

        retry_count += 1
        if retry_count <= max_retries:
            sleep_time = 2 ** retry_count
            time.sleep(sleep_time)

    return result
```

### 5.2. セキュリティ検証ロジック

**検証フロー**:
```python
def validate_auto_execute_task(task: Task, allowed_tasks: dict) -> bool:
    # 1. taskKey存在確認
    task_key = task["autoExecute"]["taskKey"]
    if task_key not in allowed_tasks:
        raise SecurityError(f"taskKey '{task_key}' not in allowed_tasks.yml")

    # 2. Task ID許可確認
    allowed_ids = allowed_tasks[task_key].get("allowedTaskIds", [])
    if allowed_ids and not match_task_id(task["id"], allowed_ids):
        raise SecurityError(f"Task ID '{task['id']}' not allowed")

    # 3. コマンドインジェクション検証
    command = allowed_tasks[task_key]["command"]
    if contains_injection_patterns(command):
        raise SecurityError(f"Command injection detected")

    return True
```

**ワイルドカードマッチング**:
```python
def match_task_id(task_id: str, patterns: List[str]) -> bool:
    """
    例:
    - task_id="1.1.1", patterns=["*"] → True
    - task_id="1.1.1", patterns=["1.*"] → True
    - task_id="1.1.1", patterns=["1.1.*"] → True
    - task_id="1.1.1", patterns=["1.1.1"] → True
    - task_id="1.1.1", patterns=["2.*"] → False
    """
    for pattern in patterns:
        if pattern == "*":
            return True
        if pattern.endswith(".*"):
            prefix = pattern[:-2]
            if task_id == prefix or task_id.startswith(prefix + "."):
                return True
        elif task_id == pattern:
            return True
    return False
```

**コマンドインジェクション検出**:
```python
INJECTION_PATTERNS = [
    r'\$\{?\w+\}?',  # 環境変数: $VAR, ${VAR}
    r'[|;&><]',      # パイプ・リダイレクト: |, &&, ;, >, <
    r'`.*?`',        # バッククォート: `command`
    r'\$\(.*?\)',    # サブシェル: $(command)
]

def contains_injection_patterns(command: str) -> bool:
    return any(re.search(pattern, command) for pattern in INJECTION_PATTERNS)
```

## 6. データモデル仕様

### 6.0. autoExecuteフィールドパターン（Pattern B推奨）

#### パターンの定義

**Pattern B（推奨）: 明示的enabled指定**
- 手動タスク: `"autoExecute": {"enabled": false}` を明示的に設定
- 自動実行タスク: `"autoExecute": {"enabled": true, ...}` で完全な設定を記述

**Pattern A（非推奨）: 暗黙的パターン**
- 手動タスク: `autoExecute`フィールドなし
- 自動実行タスク: `autoExecute`フィールドあり
- リスク: フィールド追加忘れに気づきにくい

#### Pattern B 実装例

**手動タスクの例**:
```json
{
  "id": "1.2.3",
  "title": "週次進捗レポート作成",
  "description": "Weekly progress report compilation",
  "status": "pending",
  "priority": "medium",
  "dependencies": [],
  "autoExecute": {
    "enabled": false  // 明示的に自動実行を無効化
  }
}
```

**自動実行タスクの例**:
```json
{
  "id": "3.2.1",
  "title": "Nightly Integration Test Run",
  "description": "Run integration tests every night at 3:00 AM",
  "status": "pending",
  "priority": "high",
  "dependencies": ["3.1.2"],
  "autoExecute": {
    "enabled": true,
    "taskKey": "run_integration_tests",
    "schedule": "nightly",
    "retryOnFailure": true,
    "maxRetries": 2,
    "lastExecuted": null,
    "executionHistory": []
  }
}
```

#### Pattern B 推奨理由

**メリット**:
1. **エラー防止**: フィールド追加忘れを検出可能
   - JSON Schema検証で`autoExecute`フィールドの有無をチェック可能
   - 新規タスク追加時にフィールド設定を強制できる

2. **意図明確化**: 手動実行が設計意図であることを明示
   - コードレビュー時に手動/自動の区別が一目瞭然
   - 将来の保守者が設計意図を理解しやすい

3. **移行容易性**: 後から自動実行化する際の変更が明確
   - `"enabled": false` → `"enabled": true` + 追加設定のみ
   - git diffで変更内容が明確に表示される

4. **一貫性**: すべてのタスクが同じフィールド構造を持つ
   - タスク処理ロジックが単純化される
   - フィルタリング・検索が容易

**デメリット（許容範囲）**:
- JSONファイルサイズが約7%増加（1000タスクで約70KB増）
- 手動タスクに`"enabled": false`を毎回記述する必要がある

#### Pattern A との比較

| 項目 | Pattern A（暗黙的） | Pattern B（明示的） |
|------|-------------------|-------------------|
| 手動タスク表現 | フィールドなし | `{"enabled": false}` |
| フィールド追加忘れ検出 | ❌ 不可能 | ✅ 可能（Schema検証） |
| 設計意図の明確性 | ⚠️ 曖昧 | ✅ 明確 |
| 自動実行化の変更量 | フィールド追加 | `enabled`値変更のみ |
| JSONサイズ | 100% | 107% |
| 移行リスク | 高（見落としリスク） | 低（明示的変更） |

#### 実装ガイドライン

**新規タスク作成時**:
```bash
# Task Master CLIで作成（自動的にPattern Bで生成）
task-master add-task --prompt="Weekly report creation"

# 手動でtasks.json編集する場合は、必ず以下を含める:
"autoExecute": {"enabled": false}
```

**既存タスクの移行**（Pattern A → Pattern B）:
```bash
# 全手動タスクに"autoExecute": {"enabled": false}を追加
# 実装予定: .taskmaster/scripts/migrate_to_pattern_b.py
```

**JSON Schema検証**:
```json
{
  "properties": {
    "autoExecute": {
      "type": "object",
      "required": ["enabled"],
      "properties": {
        "enabled": {"type": "boolean"}
      }
    }
  },
  "required": ["id", "title", "status", "autoExecute"]
}
```

### 7.1. タスクが自動実行されない

**症状**: 深夜帯にタスクが実行されない

   ```bash

   date  # 現在時刻確認
   ```

3. `lastExecuted` が18時間以内でないか
   ```bash
   task-master show <task-id>
   # → lastExecuted: 前回実行時刻を確認
   ```

4. 依存関係が解決済みか
   ```bash
   task-master show <task-id>
   # → dependencies: [] または全て done を確認
   ```
