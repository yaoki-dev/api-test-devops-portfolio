# Task Master 自動実行システム設計書

*最終更新: 2025年10月30日*

## 📋 エグゼクティブサマリー

### 設計の目的
Task Masterの既存タスク管理システムに、v3.0の自動実行機能を統合し、深夜帯等のオフアワーに自動でコマンド実行・ステータス更新を行う仕組みを提供する。

### 設計方針
- **単一情報源の維持**: Task Master（tasks.json）を唯一のタスク管理システムとする
- **段階的導入**: 現在自動実行タスクは0件、将来的に追加可能な拡張性を確保
- **セキュリティ優先**: v3.0のallowed_tasks.yml方式を踏襲し、コマンドインジェクションを防止
- **既存システム尊重**: Task Masterの階層構造（1.1.1形式）とMCP統合を維持

### 実装工数見積もり
- **Phase 1**: スキーマ設計・ドキュメント作成（2h）
- **Phase 2**: 実行エンジン実装（6h）
- **Phase 3**: スケジューラ統合（2h）
- **合計**: 約10時間

## 1. スキーマ設計

### 1.1. 設計選択肢

#### Option A: 専用フィールド方式（推奨）

**tasks.json拡張イメージ**:
```json
{
  "id": "1.1.1",
  "title": "Docker Volume Permission Regression Tests",
  "description": "Automated regression test for volume permissions",
  "status": "pending",
  "priority": "high",
  "dependencies": [],
  "autoExecute": {
    "enabled": true,
    "taskKey": "run_regression_docker",
    "schedule": "nightly",
    "retryOnFailure": true,
    "notifyOnCompletion": false
  }
}
```

**メリット**:
- ✅ statusワークフローと独立（pending/in-progress/done等と非干渉）
- ✅ リッチメタデータ（schedule、retry、通知設定等）を格納可能
- ✅ 将来的な拡張性が高い（実行履歴、依存関係等を追加可能）
- ✅ 既存タスクへの影響なし（autoExecuteフィールド未定義なら自動実行しない）

**デメリット**:
- ⚠️ スキーマ変更が必要（Task Masterコアへの拡張）
- ⚠️ JSON検証ロジックの追加

#### Option B: Status値方式

**tasks.json拡張イメージ**:
```json
{
  "id": "1.1.1",
  "title": "Docker Volume Permission Regression Tests",
  "status": "auto-scheduled",
  "taskKey": "run_regression_docker",
  "autoSchedule": "nightly"
}
```

**メリット**:
- ✅ スキーマ変更が最小限（既存statusフィールド活用）
- ✅ 実装コストが低い

**デメリット**:
- ❌ status値の意味が複雑化（pending/doneとauto-scheduledの関係が不明瞭）
- ❌ 実行後のステータス遷移が複雑（auto-scheduled → in-progress → done → auto-scheduled?）
- ❌ リッチメタデータ格納が困難
- ❌ 既存のステータスベースのワークフローと衝突リスク

### 1.2. 推奨設計: Option A（専用フィールド方式）

**理由**:
1. **意味の明確性**: statusは手動/AIタスクの進行状態、autoExecuteは自動実行設定
2. **拡張性**: スケジュール種別（nightly/hourly/weekly）、リトライ回数、通知先等を柔軟に追加可能
3. **保守性**: 既存のTask Masterワークフロー（task-master set-status等）と非干渉
4. **将来性**: 実行履歴（executionHistory[]）、実行ログパス等のメタデータを追加可能

### 1.3. autoExecuteフィールド詳細仕様

```typescript
interface AutoExecute {
  enabled: boolean;           // 自動実行の有効化フラグ
  taskKey: string;            // allowed_tasks.ymlのキー（必須）
  schedule: ScheduleType;     // 実行スケジュール種別
  timeWindow?: TimeWindow;    // カスタム時間帯（任意）
  retryOnFailure?: boolean;   // 失敗時リトライ（デフォルト: true）
  maxRetries?: number;        // 最大リトライ回数（デフォルト: 3）
  notifyOnCompletion?: boolean; // 完了通知（デフォルト: false）
  lastExecuted?: string;      // 最終実行日時（ISO 8601）
  executionHistory?: ExecutionRecord[]; // 実行履歴
}

type ScheduleType =
  | "nightly"    // 深夜1回（11:00-05:00 UTC+7のランダム時刻）
  | "hourly"     // 毎時1回
  | "daily"      // 毎日指定時刻
  | "weekly"     // 毎週指定曜日・時刻
  | "custom";    // timeWindow指定

interface TimeWindow {
  startHour: number;  // 実行開始時刻（0-23）
  endHour: number;    // 実行終了時刻（0-23）
  daysOfWeek?: number[]; // 実行曜日（0=日曜, 6=土曜）
}

interface ExecutionRecord {
  timestamp: string;    // 実行日時（ISO 8601）
  exitCode: number;     // 終了コード
  duration: number;     // 実行時間（秒）
  stdout?: string;      // 標準出力（オプション）
  stderr?: string;      // 標準エラー（オプション）
}
```

**具体例**:

```json
{
  "id": "3.2.1",
  "title": "Nightly Integration Test Run",
  "status": "pending",
  "autoExecute": {
    "enabled": true,

    "schedule": "nightly",
    "retryOnFailure": true,
    "maxRetries": 2,
    "notifyOnCompletion": false,
    "lastExecuted": "2025-10-29T03:15:00Z",
    "executionHistory": [
      {
        "timestamp": "2025-10-29T03:15:00Z",
        "exitCode": 0,
        "duration": 125.3
      },
      {
        "timestamp": "2025-10-28T02:47:00Z",

}
```

### 2.2. セキュリティ検証ロジック

**検証フロー**:
1. **taskKey存在確認**: allowed_tasks.ymlにtaskKeyが定義されているか
2. **Task ID許可確認**: allowedTaskIdsが設定されている場合、実行タスクIDが許可リストに含まれるか
3. **コマンド検証**: 環境変数展開・パイプ・リダイレクト等のインジェクション防止
4. **タイムアウト設定**: 必須項目、デフォルト600秒

**検証擬似コード**:
```python
def validate_auto_execute_task(task: Task, allowed_tasks: dict) -> bool:
    if not task.get("autoExecute", {}).get("enabled"):
        return False

    task_key = task["autoExecute"]["taskKey"]

    # taskKey存在確認
    if task_key not in allowed_tasks:
        raise SecurityError(f"taskKey '{task_key}' not found in allowed_tasks.yml")

    # Task ID許可確認
    allowed_ids = allowed_tasks[task_key].get("allowedTaskIds", [])
    if allowed_ids and not match_task_id(task["id"], allowed_ids):
        raise SecurityError(f"Task ID '{task['id']}' not allowed for taskKey '{task_key}'")

    # コマンドインジェクション検証
    command = allowed_tasks[task_key]["command"]
    if contains_injection_patterns(command):
        raise SecurityError(f"Command injection detected in '{command}'")

    return True

def match_task_id(task_id: str, allowed_patterns: list[str]) -> bool:
    """
    ワイルドカードパターンマッチング
    例: "1.1.1" matches "1.1.*", "1.*", "*"
    """
    import fnmatch
    return any(fnmatch.fnmatch(task_id, pattern) for pattern in allowed_patterns)
```

### 3.3. 実行フィルタリングロジック

**スクリプト起動時の判定フロー**:
```python
def should_execute_now(task: Task) -> bool:
    """
    現在時刻でタスクを実行すべきか判定
    """
    auto_exec = task.get("autoExecute", {})
    if not auto_exec.get("enabled"):
        return False

    schedule = auto_exec["schedule"]
    now = datetime.now()

    if schedule == "nightly":
        # 深夜帯（11:00-05:00 UTC+7）のみ実行
        if not (1 <= now.hour < 5):
            return False

    elif schedule == "hourly":
        # 毎時実行（時間帯制限なし）
        pass

    elif schedule == "custom":
        # カスタム時間帯チェック
        time_window = auto_exec.get("timeWindow", {})
        start = time_window.get("startHour", 0)
        end = time_window.get("endHour", 24)
        days = time_window.get("daysOfWeek")

        if not is_in_time_window(now, start, end, days):
            return False

    # 最終実行時刻チェック（重複実行防止）
    last_executed = auto_exec.get("lastExecuted")
    if last_executed:
        last_time = datetime.fromisoformat(last_executed)
        if schedule == "nightly" and (now - last_time).total_seconds() < 18 * 3600:
            # nightlyは18時間以内の再実行を防止
            return False

    return True
```

### 4.3. Task Master MCP統合

**ステータス更新方式**:

```python
import subprocess
import json

def update_task_status(task_id: str, status: str) -> bool:
    """
    Task Master MCPを使ってタスクステータスを更新
    """
    try:
        # task-master CLIを使用（MCPサーバーなしで直接実行）
        result = subprocess.run(
            ["task-master", "set-status", f"--id={task_id}", f"--status={status}"],
            capture_output=True,
            text=True,
            timeout=10
        )

        if result.returncode == 0:
            print(f"✅ Task {task_id} status updated to '{status}'")
            return True
        else:
            print(f"❌ Failed to update task {task_id}: {result.stderr}")
            return False

    except Exception as e:
        print(f"❌ Error updating task {task_id}: {e}")
        return False
```

**実行履歴更新方式**:

```python
def update_execution_history(task_id: str, execution_record: dict) -> bool:
    """
    tasks.jsonのexecutionHistory[]を直接更新
    """
    tasks_json_path = ".taskmaster/tasks/tasks.json"

    with open(tasks_json_path, "r") as f:
        data = json.load(f)

    # タスク検索（階層構造対応）
    task = find_task_by_id(data["tasks"], task_id)
    if not task:
        return False

    # executionHistory追加
    if "autoExecute" not in task:
        task["autoExecute"] = {}

    if "executionHistory" not in task["autoExecute"]:
        task["autoExecute"]["executionHistory"] = []

    task["autoExecute"]["executionHistory"].insert(0, execution_record)

    # 最新10件のみ保持（古い履歴は削除）
    task["autoExecute"]["executionHistory"] = task["autoExecute"]["executionHistory"][:10]

    # lastExecuted更新
    task["autoExecute"]["lastExecuted"] = execution_record["timestamp"]

    # ファイル書き込み
    with open(tasks_json_path, "w") as f:
        json.dump(data, f, indent=2, ensure_ascii=False)

    return True

def find_task_by_id(tasks: list, task_id: str) -> dict | None:
    """
    階層構造を再帰的に検索してTask IDに一致するタスクを返す
    """
    for task in tasks:
        if task["id"] == task_id:
            return task
        if "subtasks" in task:
            result = find_task_by_id(task["subtasks"], task_id)
            if result:
                return result
    return None
```

## 5. エラーハンドリング・リトライ戦略

### 5.1. エラー種別と対応

| エラー種別 | 検出方法 | 対応 | リトライ |
|-----------|---------|-----|---------|
| **タイムアウト** | subprocess timeout | status="error" | あり（maxRetries以内） |
| **コマンド失敗** | exit_code != 0 | status="error" | あり（maxRetries以内） |
| **セキュリティ違反** | taskKey未定義 | スキップ、エラーログ | なし |
| **依存関係未解決** | dependencies not done | スキップ、次回再試行 | 暗黙的（次回実行時） |
| **JSON破損** | tasks.json parse error | 緊急停止、通知 | なし |

### 5.2. リトライロジック

```python
def execute_with_retry(task: Task, allowed_tasks: dict) -> ExecutionResult:
    """
    リトライ機能付き実行
    """
    auto_exec = task["autoExecute"]
    max_retries = auto_exec.get("maxRetries", 3)
    retry_count = 0

    while retry_count <= max_retries:
        try:
            result = execute_command(task, allowed_tasks)

            if result.exit_code == 0:
                # 成功
                return result

            # 失敗時、リトライ判定
            if not auto_exec.get("retryOnFailure", True):
                return result  # リトライ無効

            retry_count += 1
            if retry_count <= max_retries:
                sleep_time = 2 ** retry_count  # 指数バックオフ
                print(f"⚠️ Retry {retry_count}/{max_retries} after {sleep_time}s...")
                time.sleep(sleep_time)

        except subprocess.TimeoutExpired:
            retry_count += 1
            if retry_count > max_retries:
                return ExecutionResult(exit_code=-1, error="Timeout after retries")

    return ExecutionResult(exit_code=-1, error="Max retries exceeded")
```

### 8.2. トラブルシューティング

#### 問題1: タスクが自動実行されない

**確認事項**:
1. `autoExecute.enabled` が `true` か
2. `schedule` が現在時刻に一致しているか
3. `lastExecuted` が18時間以内でないか（nightlyの場合）
4. 依存関係（dependencies）が解決済みか

**デバッグ方法**:
```bash
# ドライラン実行（実際には実行せず、フィルタリングのみ）

```

#### 問題2: セキュリティ違反エラー

**エラーメッセージ例**:
```
SecurityError: taskKey 'my_task' not found in allowed_tasks.yml
```

**解決方法**:
1. allowed_tasks.ymlに該当taskKeyが定義されているか確認
2. タイポ・大文字小文字の確認
3. allowedTaskIdsでTask IDが許可されているか確認

#### 問題3: タイムアウト頻発

**解決方法**:
1. allowed_tasks.ymlのtimeout値を増やす
2. コマンドの実行時間を短縮（テスト範囲の縮小等）
3. maxRetriesを増やす（一時的な負荷対策）

## 9. セキュリティ考慮事項

### 9.1. コマンドインジェクション防止

**禁止パターン**:
- 環境変数展開: `$VAR`, `${VAR}`
- シェルパイプ: `|`, `&&`, `;`
- リダイレクト: `>`, `>>`, `<`
- サブシェル: `` `command` ``, `$(command)`

**検証関数**:
```python
import re

INJECTION_PATTERNS = [
    r'\$\{?\w+\}?',  # 環境変数
    r'[|;&><]',      # パイプ・リダイレクト
    r'`.*?`',        # バッククォート
    r'\$\(.*?\)',    # サブシェル
]

def contains_injection_patterns(command: str) -> bool:
    return any(re.search(pattern, command) for pattern in INJECTION_PATTERNS)
```

### 9.2. ファイルパーミッション

**推奨設定**:
- `allowed_tasks.yml`: 644 (rw-r--r--)
- `auto_executor.py`: 755 (rwxr-xr-x)
- `tasks.json`: 644 (rw-r--r--)
- `logs/*.log`: 644 (rw-r--r--)

### 9.3. アクセス制御

- **ユーザー実行**: 開発者のユーザーアカウントのみ
- **sudo禁止**: 自動実行スクリプトはroot権限不要
- **Network Isolation**: 外部ネットワークアクセスが不要なタスクは制限を検討
